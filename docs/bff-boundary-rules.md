# BFF 경계 규칙

이 문서는 BFF 마이그레이션을 위한 데이터 경로 기준을 정의합니다.

## 라우팅 기준
- BFF 라우트는 `src/app/api` 아래에 둡니다. (필요할 때만 생성)
- 리소스 중심 경로를 우선합니다. (예: `/api/posts/:id`, `/api/posts/:id/likes`)

## 라우트 네이밍 규칙
- 리소스는 복수형을 사용합니다. (예: `/api/posts`, `/api/comments`)
- 서브 리소스는 하위 경로로 표현합니다. (예: `/api/posts/:id/likes`)
- 동사형 경로는 인증/액션처럼 불가피한 경우에만 사용합니다. (예: `/api/auth/reissue`)

## 데이터 경로 규칙
- 인증/세션 및 모든 쓰기 요청은 Server Action을 기본 경로로 사용합니다.
- BFF(`/api/*`)는 외부 호출, 자동 재발급 등 HTTP 엔드포인트가 필요한 경우에만 사용합니다.
- 개인화 읽기(마이페이지, 내 활동 등)는 서버 전용 모듈(RSC/Server)을 우선하고, 필요할 때만 BFF를 사용합니다.
- 공개 읽기는 RSC를 우선하며, 직접 호출은 명확한 근거가 있을 때만 허용합니다.
- 파일 업로드는 예외로 `/api` BFF 경로를 사용합니다. (클라이언트 업로드 훅과 호환성 유지)

## 캐시 무효화 최소 규칙 (CUD 공통)
- CUD Server Action은 **반드시 서버에서 캐시 무효화를 처리**합니다.
- 게시글 관련 CUD는 `postCacheInvalidation`만 사용합니다. (직접 `revalidatePath` 호출 금지)
- 클라이언트에서 `revalidatePostAction`을 호출하지 않습니다. (필요 시 제거)
- 태그/경로 기준은 `posts-list`, `post-{id}`를 고정 사용합니다.

## 응답 포맷
- 표준 응답 형태는 `{ success, data, error }` 입니다.
- 에러 객체는 안정적인 `code`와 사용자용 `message`를 포함합니다.

## 요청 규칙
- 요청 본문은 기본적으로 JSON을 사용합니다.
- 파일 업로드는 `FormData`를 사용하며 `Content-Type`은 수동으로 지정하지 않습니다.
- 서버에서 백엔드 호출 시 `Authorization` 헤더는 서버에서만 주입합니다.

## 업로드 에러 메시지 규칙
- 업로드 실패 시 백엔드 응답의 `message`를 우선 사용합니다.
- `message`가 없으면 기본 메시지(`이미지 업로드에 실패했습니다.`)로 처리합니다.

## HTTP 상태 코드 규칙
- 성공: `200`(조회/일반 성공), `201`(생성), `204`(삭제)
- 인증/권한: `401`(미인증), `403`(권한 없음)
- 클라이언트 오류: `400`(요청 오류), `404`(리소스 없음)
- 서버 오류: `500`(내부 오류)
- 원칙적으로 백엔드 상태 코드를 보존하고, BFF 검증 실패만 `400`으로 처리합니다.

## 인증 예외 처리 규칙
- `401`은 **미인증**으로 처리하고, 클라이언트의 인증 캐시를 정리한 뒤 로그인 화면으로 유도합니다.
- `401` 재발급 실패 시에는 로그아웃 흐름으로 정리합니다.
- `403`은 **권한 없음**으로 처리하고, 세션은 유지한 채 권한 안내 화면/메시지를 노출합니다.
- BFF는 `401/403` 상태 코드를 그대로 전달하고, 프론트는 공통 처리 지점을 통해 일관되게 처리합니다.

## 캐시 정책
- 개인화 데이터는 기본 `no-store`를 적용합니다.
- 공개 데이터는 `revalidate` 또는 `revalidateTag`를 기준으로 설계합니다.

## 클라이언트 사용 규칙
- 마이그레이션 이후 클라이언트 컴포넌트에서 백엔드를 직접 호출하지 않습니다.
- 클라이언트 호출은 Server Action을 우선하고, 필요한 경우에만 `/api/*`를 사용합니다.
